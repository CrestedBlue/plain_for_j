# Plan: 여행 일정 저장 웹페이지 (Travel Planner)

**작성일**: 2026-06-16
**상태**: 기획 확정 — 구현 시작 직전 (Kakao 키 확인 대기)
**복잡도**: Medium

## 1. 요구사항 정리

여행 계획을 저장·열람하는 웹앱. **여러 여행묶음**을 저장하고 한 번에 하나를 관리.

| # | 동작 | 설명 |
|---|------|------|
| R0 | 여행묶음 관리 | 첫 화면에서 여행묶음 목록을 보고, 선택하거나 새로 생성. 한 번에 하나의 묶음을 작업 |
| R1 | 여행 일자 선택 | 새 묶음 생성 시 시작일~종료일 선택 → 그 범위의 날짜들이 나열됨 |
| R2 | 날짜 클릭 → 하루 일정 표시 | 선택한 날짜 아래에 그 날의 일정 리스트 (예: "여의도 방문") |
| R3 | 일정 리스트 옆 지도 | 그 날 일정에 저장된 위치들이 Kakao 지도에 마커로 표시 |
| R4 | 일정 선택 → 상세 메모 + 지도 active | 일정 하나를 고르면 상세 메모 표시 + 지도 마커 활성화(강조/중심 이동) |
| R5 | 저장 | localStorage에 저장 → 재방문 시 모든 여행묶음 유지 |

### 확정된 결정 (사용자 확인 완료)
- **지도/지오코딩**: **Kakao Map** (한국 중심, 한글 주소 정확도 우선)
- **저장**: **localStorage** (백엔드 없음, 로컬 우선)
- **여행묶음**: **여러 개 저장**, 첫 화면에서 선택/생성, **한 번에 하나** 관리
- **위치 입력**: **검색(키워드/주소) + 지도 클릭 핀** 둘 다 지원

## 2. 기술 스택

| 영역 | 선택 | 비고 |
|------|------|------|
| 빌드/프레임워크 | Vite + React + TypeScript | 가벼운 SPA |
| 지도 | **Kakao Maps JavaScript SDK** | 마커/active 제어 |
| 위치 검색 | **Kakao Maps `services` 라이브러리** (`Places` 키워드검색, `Geocoder` 주소/좌표변환) | 클라이언트에서 직접 호출, 백엔드 불필요 |
| 지도 클릭 핀 | Kakao `click` 이벤트 + `Geocoder.coord2Address` (역지오코딩) | 좌표→장소명 |
| 상태관리 | Zustand + `persist` 미들웨어 | localStorage 영속화 |
| 스타일 | CSS + 디자인 토큰(CSS custom properties) | web 룰 준수 |
| 테스트 | Vitest(단위) + Playwright(E2E) | web 룰 준수 |

> ⚠️ **선행 필요 — Kakao 앱 키**: Kakao Developers에서 앱 생성 → **JavaScript 키** 발급 → 플랫폼에 사이트 도메인 등록(`http://localhost:5173` 개발용). 키는 `.env`의 `VITE_KAKAO_MAP_KEY`로 주입(코드 하드코딩 금지). 키 없이도 스캐폴드는 가능하나 지도는 키 주입 후 동작.

## 3. 설계

### 3.1 데이터 모델

```ts
type AppState = {
  trips: Trip[];               // 여행묶음 여러 개
  activeTripId: string | null; // 현재 작업 중인 묶음
};

type Trip = {
  id: string;
  title: string;               // 예: "도쿄 3박 4일"
  startDate: string;           // 'YYYY-MM-DD'
  endDate: string;
  days: Day[];                 // 날짜 범위로부터 생성
};

type Day = {
  date: string;                // 'YYYY-MM-DD'
  items: ScheduleItem[];
};

type ScheduleItem = {
  id: string;
  title: string;               // 예: "여의도 방문"
  location?: {                 // 검색 또는 지도클릭으로 설정
    name: string;              // 장소명/주소
    lat: number;
    lng: number;
  };
  memo: string;                // 상세 메모
  order: number;               // 하루 내 정렬
};
```

### 3.2 화면 흐름

```
[첫 화면 — 여행묶음 목록 (R0)]
┌─────────────────────────────┐
│  내 여행                      │
│  ┌──────────┐ ┌──────────┐   │
│  │도쿄 3박4일│ │부산 1박2일│   │  ← 선택 시 해당 묶음 열기
│  └──────────┘ └──────────┘   │
│  [+ 새 여행 만들기]            │  ← 기간 선택(R1) → 묶음 생성
└─────────────────────────────┘
              │ 선택/생성
              ▼
[묶음 작업 화면]
┌───────────────────────────────────────────────┐
│  ← 목록   [도쿄 3박 4일]      6/20 ~ 6/23        │  ← 헤더(목록 복귀)
├───────────────────────────────────────────────┤
│  [6/20] [6/21] [6/22] [6/23]                    │  ← 날짜 탭(R2)
├──────────────────────┬────────────────────────┤
│  하루 일정 리스트      │                         │
│  • 여의도 방문  ←선택  │     Kakao 지도          │
│  • 한강 피크닉         │     ● 활성 마커          │  ← R3
│  • 저녁 식사           │     ○ ○                  │
│  [+ 일정 추가]         │   (지도 클릭 → 핀 추가)   │
├──────────────────────┴────────────────────────┤
│  상세 메모: "여의도 방문 — 11시 도착, ..."        │  ← R4
└───────────────────────────────────────────────┘
```

일정 선택 ↔ 지도 마커 active는 **양방향 동기화**.

## 8. 향후 확장 기능 (백로그 — 미착수)

> 2026-06-17 추가. 아직 구현하지 않음. 착수 전 기술 검토용 메모.
> 공통 선결조건: ① 실제 위경도(lat/lng) — Kakao 지도 연동 필요(현재 SVG용 x/y만 보유), ② 외부 API 키·CORS — 일부는 서버리스 프록시 필요(현 무백엔드 정책 재검토 지점), ③ 호출 결과는 localStorage 캐싱으로 쿼터 절약.

### F1. 일정 추천 (근처 맛집/관광 추천)
- **트리거**: 일정 하나 선택(조회) 시 "일정 추천" 버튼 노출 → 그 위치 근처 추천.
- **데이터**: Kakao Local `Places.categorySearch` (카테고리코드 `FD6`음식점·`CE7`카페·`AT4`관광명소 + 중심 좌표 x/y + radius). **services 라이브러리로 클라이언트 호출 가능 → 무백엔드 OK**.
- **UI**: 추천 리스트 → 탭하면 그 장소를 새 일정으로 추가(좌표 자동).
- **선결**: 선택 일정의 **lat/lng 필요** → 실제 Kakao 연동 의존.

### F2. 날씨 확인 (배경 + 애니메이션)
- **범위**: 예보 가능 기간(대략 향후 1~2주)만. 그 밖 날짜는 중립 배경.
- **데이터**: 기상청 단기예보(공공데이터포털) 또는 OpenWeather. 좌표(여행 대표 위치)+날짜 필요. **API 키 서버보관·CORS 이슈 → 서버리스 프록시 권장**.
- **UI**: 일자 선택 시 배경에 날씨 표현 — **비/눈/햇살/구름** CSS 애니메이션(컴포지터 친화: transform/opacity).
- **고려**: 예보 지평 제한, 대표 좌표 산정(첫 일정 또는 도시 중심), 날짜별 캐싱.

### F3. 영업중 여부
- **데이터(난점)**: 영업시간 출처가 핵심. Kakao 기본 Local API엔 영업시간 미제공. 후보 — Google Places Details `opening_hours/open_now`(키·과금·서버), 또는 수동 입력. 
- **로직**: 장소 영업시간 ↔ **선택 일자의 요일(+공휴일, F4 연계)** 대조 → 영업중/영업종료/정보없음.
- **UI**: 일정 카드/조회에 상태 배지.
- **선결**: lat/lng + place id, 외부 키.

### F4. 공휴일 입력 (Open 캘린더)
- **데이터**: 한국 우선 — 공공데이터포털 "특일정보"(`getHolDeInfo`) 또는 **Nager.Date 공개 API(무키·CORS 양호·다국가)**. 여행 연도/국가로 조회.
- **UI**: 캘린더에 공휴일 표시(예: 빨강 + 명칭). **F3 영업 판정에 입력**으로 연결.
- **확장**: 지금은 **한국만**. 해외 확장 시 — 여행묶음에 국가코드 부여 → Nager.Date `countryCode`로 동시 조회. (그때 설계)

### F5. "스크럼" — 카테고리별 장소 라이브러리 (지도 중심) 〔2026-06-17 추가〕
- **개념**: 여행과 **독립된 나만의 장소 풀(보관함)**. 지금 카테고리(식사/카페/관광/숙소 등)는 고정 라벨일 뿐 → 사용자가 **카테고리별로 실제 장소 데이터를 생성·저장**하는 별도 페이지가 필요.
- **이름/진입**: 탭 이름 **"스크럼"**. 탭 진입 시 **지도가 메인**, 검색·지도클릭으로 장소를 담아 **식사/카페/관광/숙소** 등 카테고리로 분류 저장.
- **기본 추천**: 진입 시 카테고리별 **추천 장소를 기본 제공**(예: 기존 `SEOUL_LANDMARKS` 활용 또는 큐레이션 리스트), 사용자가 추가/삭제.
- **연계**: 저장된 장소는 **일정 추가 시 라이브러리에서 선택**(장소명·좌표 자동 입력)하고, **F1 추천**의 후보 소스로도 사용.
- **데이터/구조**: 새 store `placeStore`(persist, 키 `plan-for-j/places`) + `SavedPlace { id, name, category, lat/lng 또는 x/y, memo? }`. App에 **상위 네비게이션(여행 ↔ 스크럼)** 신설 필요.
- **선결**: 지도 중심이라 실제 Kakao 연동(검색·좌표)과 함께 가면 효과적. 폴백 SVG로도 MVP 가능.

### 공통 영향 (착수 시)
- **데이터 모델**: `ScheduleItem`에 `businessHours`/`placeId`, `Trip`에 `countryCode`, 날씨/공휴일 캐시, 별도 `SavedPlace` 라이브러리(F5) 추가 가능.
- **아키텍처**: F2·F3가 키/CORS로 **서버리스 함수**(예: Vercel Functions)를 요구 → 무백엔드 정책 재검토 트리거. F1·F4·F5는 클라이언트로 가능. F5는 **상위 탭 네비게이션** 도입이 선행.
- **우선순위 제안**: 실제 Kakao 연동 → F5(장소 라이브러리, 일정·추천의 토대) → F1(추천) → F4(공휴일) → F2(날씨, 프록시) → F3(영업, 데이터 출처 확정 후).
